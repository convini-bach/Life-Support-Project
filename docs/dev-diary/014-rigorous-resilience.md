# 開発日記 014: 厳格なレジリエンス — 信頼を支えるディテール

## 概要
本日、私たちは開発の後半を「徹底的な不具合の撲滅」に捧げた。CSV エクスポートの引用符（クォート）処理、JSON 出力の MIME タイプ、そして無反応に見えた決済ボタンのデバッグ。表面的なきらびやかさよりも、押したときに動く、出したファイルが読める、という「当たり前」の品質を優先した。

## 議論と洞察のポイント

### 1. 「反応しない」という静かなエラー
ボタンが「何も起きない」というのは、ユーザーにとって最も大きな不安を与える。API がプレーンテキストでエラーを返し、フロントエンドが JSON としてパースしようとして失敗する。この静かな衝突を解決するため、すべての API レスポンスを JSON に統一し、フロントエンドでの詳細なエラー通知機能を強化した。失敗の理由を伝えることは、成功への道筋を共有することである。

### 2. CSV の「方言」と BOM
日本語環境の Excel という、独自の「方言」を持つ CSV に対し、BOM(\uFEFF) の付与と、すべてのセルをダブルクォートで保護する処理を追加した。カンマを含む料理名でもデータが崩れない。Local-First なデータ駆動型アプリにおいて、データの可搬性は生命線である。

### 3. 日付という名の共通言語
`toISOString()` に依存せず、日本標準時（JST）ベースの `YYYY-MM-DD` 取得関数を自作した。サーバーの都合ではなく、ユーザーの生活圏の時計に合わせる。利用回数管理やログの整合性を守るための、目立たないが重要な「時計合わせ」である。

## 結論
技術的なディテールへのこだわりは、ユーザーへの敬意の表れである。本日のレジリエンス強化により、Nutri-Vision は「試作品」から「実用品」としての強さを手に入れた。
